> 版本：2025-05-04
> 作者：Cascade AI + USER

---

## 0. 文件目的
1. **列出所有可預見風險，無死角。**
2. **定義量化方法 (FMEA) 以便排序處理。**
3. **對應解法/模組，形成閉環。**
4. **提供『長期維護機制』，讓後續新增/調整風險時不須大規模重寫。**

## 2. FMEA 量化模型

| 指標 | 說明 | 取值 (1–10) |
|------|------|-------------|
| 🔥 Impact (I) | 影響嚴重度 | 1=可忽略；10=劇情崩潰/用戶流失 |
| 💣 Likelihood (L) | 發生機率 | 1=罕見；10=高頻重現 |
| 🧭 Detectability (D) | 系統可偵測性 | 1=容易發現；10=不易察覺 |
| ☢️ **RPN = I×L×D** | 風險優先數 | ≥200 需立即處理 |

> 每項風險以 `risks/fmea.csv` 管理：
> `category,sub_category,impact,likelihood,detectability,rpn,mitigation_module,status`

## 4. 長期維護策略（系統化）

### 4.1 風險資料 → **Config as Data**
* 所有風險項目存 `.yaml` 或 `.csv`，與程式碼分離。
* CI 驗證：
  * schema 驗證 (e.g., `jsonschema`).
  * 重複 ID、空欄位檢查。

### 4.2 插拔式 MemoryStore
* `trait MemoryStore`（見 `src/memory/`）定義介面。
* 策略熱插拔：`naive_log` / `summary_v1` / `retrieval` / `hybrid`。
* `eva.toml` 指定當前策略，不需改 StorySession。

### 4.3 觀測 + 告警
* 在 `event_bus` 發佈：
  * `RiskDetected { id, rpn, context }`
* Prometheus/Grafana 監控高 RPN 項。
* 自動開 GitHub Issue 追蹤高 RPN。 

### 4.4 測試 & 試驗
* **單元測試**：各 MemoryStore 確保 `build_context()` 不超 token & 含 tag=推理。
* **A/B 測試**：`model_router` 支援流量分配至不同 MemoryStore，收集用戶留存、NPS。

### 4.5 文件自動生成
* `cargo xtask doc:risks` → 將 `yaml/csv` 轉 Markdown + Mermaid flowchart，發佈至 `docs/`。
